## 行业分析

### 市场规模与增长

AI 写作工具市场在 2025 年估值约 **19亿~35亿美元**（不同机构定义口径有差异），预计到 2033 年将达到 **79亿~100亿美元**，复合年增长率（CAGR）约 **23%~27%**。

_市场规模: 2025年约19亿-35亿美元_
_增长率: CAGR 23%-27%，预计2033年达79亿-100亿美元_
_AI创意内容市场（更广口径）: 2025年26.5亿美元，预计2035年达160亿美元，CAGR 19.69%_

关键增长驱动力：
- 生成式 AI 技术爆发式发展（大语言模型能力持续提升）
- 数字内容需求激增（80% 博主和 70% 以上企业已集成 AI 写作工具）
- 中国以 **67%** 的 AI 写作采用率全球领先，政府推动 + 电商/出版广泛集成

_Source: [Verified Market Research](https://www.verifiedmarketresearch.com/product/ai-writing-tool-market/), [HTF Market Intelligence](https://www.htfmarketintelligence.com/report/global-ai-writing-tool-market), [All About AI](https://www.allaboutai.com/resources/ai-statistics/ai-writing/)_

### 市场结构与细分

AI 写作工具市场可分为两大阵营：

**1. 通用型 AI 聊天助手**（ChatGPT、Claude、Gemini）
- 擅长：头脑风暴、对话生成、行级编辑
- 限制：长篇上下文管理薄弱，缺乏专业写作工作流

**2. 专业小说写作平台**（Sudowrite、Novelcrafter、NovelAI、DreamGen、Squibler）
- 擅长：长篇叙事管理、角色追踪、场景结构、连续性保持（支撑 8 万字以上的手稿）
- 差异化价值：从单纯文本生成转向结构化写作环境

**细分方向：**
- 按部署方式：云端方案主导市场
- 按应用场景：营销内容 (73%) > 媒体 (65%) > 技术 (62%)；但创意小说写作是增长最快的细分市场
- 按地区：北美占据 38% 收入份额，但中国采用率最高

_Source: [SNS Insider](https://www.snsinsider.com/reports/ai-powered-content-creation-market-8195), [The Business Research Company](https://www.thebusinessresearchcompany.com/report/ai-powered-content-creation-global-market-report)_

### 行业趋势与演变

**趋势1：AI 成为"共同作者"而非替代者**
2024 年 Authors Guild 调查显示，82% 受访作者担忧 AI 会同质化文学风格。但研究表明，只有被动使用 AI（不编辑输出）时这种担忧才成立。2026 年的最佳实践是"AI 辅助 + 人类主导"的协作模式。

**趋势2：专业化写作模型崛起**
- Sudowrite 的 **Muse** 模型：专为创意写作微调的 LLM，理解场景调度、潜台词、节奏
- NovelAI 的 **Erato 70B**：在文学语料（非网络内容）上训练的创意模型
- DreamGen 的自研模型：专为无审查叙事定制

**趋势3：多模型堆栈成为标配**
Sudowrite 同时集成 GPT-4.1、Claude Opus、Gemini Pro 2.5 等多个模型，加上自有叙事处理层。用户可按场景切换模型——创意探索用高温度模型，一致性检查用逻辑型模型。

**趋势4：从文本工具到写作操作系统**
2026 年趋势明确：不再是单一文本生成器，而是集成了知识库（Story Bible/Codex）、可视化规划（Canvas）、角色管理、连续性追踪、风格匹配的**写作操作系统**。

**趋势5：上下文管理成为核心竞争力**
"先投入上下文，再生成散文"成为 2026 年首要最佳实践。Story Bible / Codex 是维持长篇角色和情节一致性的最关键功能。

**趋势6：分析型 AI 与生成型 AI 融合**
Authors.ai 的 Marlowe 开创了分析型路线——不生成文本，而是像"读过几千本畅销书的顾问"一样分析你的手稿的节奏、角色弧线、主题、情节节拍，并与畅销书基准对比。2026 年趋势是将此类分析能力融入生成式写作平台中。

_Source: [Kindlepreneur](https://kindlepreneur.com/sudowrite-review/), [ManuscriptReport](https://manuscriptreport.com/blog/ai-publishing-trends), [Authors.ai](https://authors.ai/introducing-marlowe-3-0-and-the-author-dashboard/), [DreamGen](https://dreamgen.com/blog/articles/best-ai-story-generators)_

- **创新压力**：极高。新模型每月发布，用户期望快速迭代
- **差异化方向**：从"谁的 AI 更好写"转向"谁的写作工作流更智能"

## 竞争格局深度分析

### 关键玩家功能深度拆解

#### 1. Sudowrite — 创意写作搭档（标杆产品）

**核心产品理念**：发现式写作的 AI 搭档。操作原则是"意外之喜与增强"——你有一个场景或模糊想法，它变出散文。

**旗舰功能矩阵：**

| 功能 | 描述 | 对 WriteTeam 的参考价值 |
|------|------|----------------------|
| **Story Engine 3.0** | 输入"脑暴转储"、类型、风格、角色列表 → 自动生成章节节拍和完整散文 | ⭐⭐⭐ 全书生成工作流 |
| **Muse 模型** | 自研创意写作 LLM，理解场景调度、潜台词、节奏、对话；近乎无审查 | ⭐⭐⭐ 专业写作模型的价值证明 |
| **My Voice (Beta)** | 用作者 ≥1000 字样本私训风格模型，可在 Draft 和 Rewrite 中使用；支持强度滑块调节 | ⭐⭐⭐ 风格匹配是核心需求 |
| **Canvas 2.0** | 空间化头脑风暴板，AI 能"读取"卡片邻近关系生成情节关联；支持协作 | ⭐⭐ WriteTeam 已有 Canvas |
| **Story Bible** | 自动编目角色、世界观、情节点、时间线；Synopsis 字段扩至 4000 字；可折叠区块 | ⭐⭐⭐ 直接对标 WriteTeam 故事圣经 |
| **Write（续写）** | "类固醇版自动完成"，分析角色/基调/情节弧线，建议下 300 字；结果显示在右侧历史栏 | ⭐⭐⭐ 对应"幽灵文字"愿景 |
| **Describe** | 选中文本 → 生成感官描写、隐喻、风格修饰；多种感官维度可选 | ⭐⭐⭐ 修辞增强工具 |
| **Expand** | 将摘要式短文本扩展为完整场景，考虑前文上下文 | ⭐⭐ WriteTeam 已有扩写 |
| **Brainstorm** | 分类式灵感生成（对话/角色/世界观/名字等）；👍👎学习偏好 | ⭐⭐ 反馈学习机制值得参考 |
| **Shrink Ray** | 将整章浓缩为节拍表（Beat Sheet），逆向工程分析节奏 | ⭐⭐ 节奏分析工具 |
| **Sensory Rewrite** | 高亮段落 → 按指定感官（视/嗅/触）或作家风格重写 | ⭐⭐ 风格化改写 |
| **多模型堆栈** | GPT-4.1, Claude Opus, Gemini 3.1 Pro, Grok 4, DeepSeek, Muse 等 | ⭐⭐ WriteTeam 已有 BYOK |
| **POV & Tense 控制** | 设定一次，所有生成保持一致的人称和时态 | ⭐⭐⭐ 中文小说人称一致性关键 |
| **Text Highlighting** | 6 色高亮，用于追踪故事线/标记修订区域；导出时保留 | ⭐ 基础功能 |
| **Chapter Continuity** | 章节可链接，AI 使用前文上下文保持弧线连续 | ⭐⭐⭐ 长篇连续性核心 |

**UX 设计哲学**：极简界面，功能通过侧栏和工具栏暴露。AI 建议显示在右侧"History"栏，用紫色标记 AI 生成文本。核心工作流：Write → Generate → Curate → Edit → Repeat。

_Source: [Sudowrite Docs](https://docs.sudowrite.com/using-sudowrite/1ow1qkGqof9rtcyGnrWUBS/my-voice-beta/5yjethEvdaYgJc1Ss3ExG5), [Kindlepreneur Review](https://kindlepreneur.com/sudowrite-review/), [NerdyNav Review](https://nerdynav.com/sudowrite-review/), [AI Tools DevPro](https://aitoolsdevpro.com/ai-tools/sudowrite-guide/)_

#### 2. Novelcrafter — 结构化小说工坊（最强知识库）

**核心产品理念**："大师蓝图"。假设你要构建一个有规则的世界、有历史的角色、有后果的情节。

**旗舰功能矩阵：**

| 功能 | 描述 | 对 WriteTeam 的参考价值 |
|------|------|----------------------|
| **Codex** | 结构化知识库：角色/地点/物品/传说/子情节/自定义类别；自定义字段；AI 自动注入相关条目到 Prompt | ⭐⭐⭐ 最强知识库设计，WriteTeam 应深度学习 |
| **Progressions** | 记录角色随时间变化（年龄/关系/世界政治变化）——动态而非静态笔记 | ⭐⭐⭐ 角色成长弧线追踪 |
| **Smart Highlighting** | 自动标记弱点：过度使用的词、拐杖短语、重复隐喻、AI 生成模式 | ⭐⭐⭐ 质量分析工具 |
| **Plan Interface** | 网格/矩阵/大纲三种视图；拖放式 Acts/Chapters/Scenes 组织 | ⭐⭐ 结构化规划 |
| **Scene Summaries** | 场景摘要自动作为后续场景的 AI 上下文（系统 Prompt） | ⭐⭐⭐ 上下文传递机制核心参考 |
| **Scene Beats** | "/" 快捷键插入 Beat（~500 字的指令），一个场景通常 6+ Beats | ⭐⭐⭐ 精细化场景控制 |
| **Custom Prompts** | 所有 Prompt 可克隆/修改/新建；可查看发送给 AI 的精确 Prompt | ⭐⭐⭐ Prompt 透明化 |
| **AI Chat + Codex** | 与小说聊天、采访角色（Chat 自动加载 Codex 记忆） | ⭐⭐ WriteTeam 已有 AI Chat |
| **Story Timeline** | 右侧垂直时间线，概览全书场景分布 | ⭐⭐ 可视化导航 |
| **BYOK 300+ 模型** | OpenRouter / 本地模型（LM Studio, Ollama）全支持 | ⭐⭐ WriteTeam 已有 BYOK |
| **Reasoning/Thinking** | 实验性支持 AI 推理/思考链 | ⭐⭐ 前沿功能 |
| **角色自动检测** | AI 自动检测并链接文本中的 Codex 条目引用，提供内联预览 | ⭐⭐⭐ 对应 WriteTeam Saliency |

**UX 设计哲学**：写作界面 + 侧边 Codex 面板 + 底部时间线。Focus Mode 隐藏一切只留文字。学习曲线陡峭但功能极强。被称为"AI 辅助小说的 Photoshop"。

_Source: [Novelcrafter Features](https://www.novelcrafter.com/features), [Novelcrafter Codex](https://www.novelcrafter.com/features/codex), [Novelcrafter Help](https://www.novelcrafter.com/help/getting-started/quick-start/organizing-in-novelcrafter), [Kindlepreneur Review](https://kindlepreneur.com/novelcrafter-review/)_

#### 3. Marlowe (Authors.ai) — 手稿分析顾问（分析型 AI 标杆）

**核心产品理念**：不写文字，只读你的书。像"读过几千本畅销书的编辑顾问"。

**旗舰功能矩阵：**

| 功能 | 描述 | 对 WriteTeam 的参考价值 |
|------|------|----------------------|
| **Marlowe 3.0 + Author Dashboard** | 2025年9月发布，交互式在线报告取代静态 PDF | ⭐⭐⭐ 分析型 AI 的产品形态 |
| **22 类型基准对比** | 将手稿与畅销书对比：情节/角色/节奏/主题/对话/设定等 | ⭐⭐⭐ "完稿体检报告"的直接参考 |
| **节奏模拟** | 模拟读者翻页体验，显示哪些句子快节奏/慢节奏 | ⭐⭐⭐ 对应"节奏心电图"愿景 |
| **角色弧线分析** | AI 理解角色特质和发展弧线 | ⭐⭐⭐ 角色深度分析 |
| **Story Beats 分析** | 分析故事节拍是否符合类型惯例 | ⭐⭐ 结构分析 |
| **可读性/对话比/陈词滥调检测** | 基础分析（免费版） | ⭐⭐ 文本质量检查 |

**关键设计决策**：
- **分析型而非生成型**——不与作者竞争写作，而是增强作者的判断力
- **权利清晰的训练语料**——不在用户手稿上训练
- **15 分钟出报告**——快速反馈循环

_Source: [Authors.ai](https://authors.ai/), [Marlowe Pro](https://authors.ai/marlowe-pro/), [Authors.ai Blog](https://authors.ai/introducing-marlowe-3-0-and-the-author-dashboard/)_

### 竞争定位与差异化策略

**差异化维度分析：**

```
                        结构化控制 ←————————————→ 创意自由
                              |                      |
                    Novelcrafter ●                ● Sudowrite
                              |                      |
                   Squibler ●  |               ● NovelAI
                              |                      |
                    ShyEditor ●|               ● DreamGen
                              |                      |
                        分析型 ←————————————→ 生成型
                              |                      |
                     Marlowe ●|              （多数产品）
```

| 差异化策略 | 代表产品 | 核心壁垒 |
|-----------|---------|---------|
| **自研模型** | Sudowrite (Muse), NovelAI (Erato), DreamGen | 模型训练数据和微调能力 |
| **知识库深度** | Novelcrafter (Codex), Sudowrite (Story Bible) | 世界观管理的结构化设计 |

## 技术趋势与创新

### 1. 上下文窗口：从数量到质量的转变

**现状（2026年初）：**
- Claude 4 Sonnet: 200K 标准 / 1M Beta
- GPT-5.2: 400K 上下文
- Gemini: 2M 上下文
- Llama 4 Scout: 10M 上下文（行业最大）

**关键发现——"中间遗忘"问题**：
尽管上下文窗口已足够容纳整部小说（~8万字 ≈ 10万~16万 tokens），但 LLM 在中间位置的信息检索准确率显著下降（从 75% 降至 55-60%）。即使声称 200K 的模型，实际有效性通常在 ~130K 时就开始不可靠。

**多数模型有效利用率仅 10-20%**——这意味着仅凭大上下文窗口无法解决长篇小说的连续性问题。

**最佳实践：**
- 关键信息放在上下文的**开头和结尾**（对抗"中间遗忘"）
- 实施上下文轮换（Context Rotation）
- 使用 RAG 补充而非堆砌全部上下文
- 利用 Prompt Caching（可节省 90% 重复内容成本）
- **黄金法则**：从最小可用上下文开始，仅在碰到限制时扩展

**对 WriteTeam 的意义**：WriteTeam 当前的 story-context.ts 上下文编排系统方向正确——精选上下文比堆砌全文更有效。需要进一步优化为多层记忆架构。

_Source: [Claude5 Hub](https://claude5.com/news/context-window-evolution-how-200k-to-1m-tokens-redefine-ai-c), [Zylos Research](https://zylos.ai/research/2026-01-19-llm-context-management), [Local AI Zone](https://local-ai-zone.github.io/guides/context-length-optimization-ultimate-guide-2025.html)_

### 2. RAG 在创意写作中的应用

**核心模式**：将 Story Bible（世界观、角色档案、时间线、地图）索引到向量存储，LLM 检索相关信息后生成接地的回答——保持叙事一致性，减少幻觉。

**RAG vs 大上下文窗口**：
- 有观点认为"RAG 已死"——百万 token 上下文窗口使检索变得不必要
- 但实际上：上下文感知的 RAG 系统**保持 91% 关键信息同时减少 68% 上下文大小**
- 对长篇小说系列（多本书共享世界观），RAG 仍然不可替代

**前沿研究**：
- **Tracert-RAG**（MDPI 2025）：方向感知图遍历，在文学分析任务中达到 8.05/10 质量分，比标准 RAG 快 80 倍
- **Disco-RAG**（2026年1月）：利用篇章内话语树和篇章间修辞图来规划和编排生成，大幅改善长文本/半结构化输入的连贯性
- **DF-RAG**（2026年1月）：多样性感知检索，优化多跳推理的召回率

**Novelcrafter 的 Codex = 作者自建的 RAG 网络**——这是目前市场上最成功的创意写作 RAG 实现。

**对 WriteTeam 的意义**：当前 Story Bible + Saliency 系统可以升级为真正的 RAG 架构——将角色、世界观、时间线条目向量化，按场景需求智能检索注入 Prompt。

_Source: [Medium - Shadmehr Vadoodi](https://medium.com/@shadmehrv/how-retrieval-augmented-generation-rag-can-supercharge-storytelling-for-writers-3f07619b7ea4), [MDPI - Tracert-RAG](https://www.mdpi.com/2673-4591/120/1/47), [Medium - RAG is Dead](https://medium.com/@reliabledataengineering/rag-is-dead-and-why-thats-the-best-news-you-ll-hear-all-year-0f3de8c44604)_

2. **GPT-5.2**: 8.51/10 — 更擅长遵循复杂约束和精确Brief

### 4. 从 Prompt 工程到 Context 工程

**范式转变**（Andrej Karpathy, 2025年6月）：
> "LLM 是 CPU，上下文窗口是 RAM，你的工作是做操作系统——为每个任务加载恰好正确的代码和数据到工作内存。"

**核心原则**：
- **结构胜过措辞**：清晰的结构和上下文比聪明的措辞更重要——大多数 Prompt 失败来自歧义而非模型限制
- **冷静直接的指令**：激进语言（"CRITICAL!", "YOU MUST", "NEVER EVER"）在新 Claude 模型上反而降低质量
- **使用标签内容**：`<context>` 标签包裹引用内容，效果显著可测量
- **四要素框架**：Persona（角色）+ Task（任务）+ Context（上下文）+ Format（格式）

**小说写作特有洞察**：
- 通用 AI（ChatGPT/Claude 直接使用）没有 Story Bible、没有风格匹配、没有一致性追踪——每次交互都得手动 Prompt 工程，对长篇写作极其疲惫
- 专业写作平台的价值 = **将 Context 工程产品化**——让作者无需理解 Prompt 技术就能获得精准的上下文注入

**对 WriteTeam 的意义**：WriteTeam 的 `buildStoryPromptContext()` 就是 Context 工程的产品化实现。优化方向：按功能类型（写作/规划/检查）差异化上下文组装，利用 Saliency 分析动态选择最相关的角色/地点注入。

_Source: [Lakera](https://www.lakera.ai/blog/prompt-engineering-guide), [IBM](https://www.ibm.com/think/prompt-engineering), [OpenAI](https://platform.openai.com/docs/guides/prompt-engineering)_

### 5. SSE 流式架构——仍然是最佳选择

**2026 年结论**：Server-Sent Events (SSE) 仍然是 LLM 流式输出的最佳协议。

| 方案 | 优势 | 劣势 |
|------|------|------|
| **SSE** | 无状态、轻量、基于 HTTP、自动重连 | 仅单向（服务器→客户端） |
| **WebSocket** | 双向通信 | 有状态连接、连接池管理复杂、负载均衡困难 |
| **gRPC** | 强工具链 | 需紧密耦合整个技术栈 |

**Vercel AI SDK `streamText`** 是当前最成熟的前端流式消费方案——WriteTeam 当前使用的 SSE 流式管道架构方向完全正确。

_Source: [Procedure.tech](https://procedure.tech/blogs/the-streaming-backbone-of-llms-why-server-sent-events-(sse)-still-wins-in-2025), [AI SDK](https://ai-sdk.dev/docs/reference/ai-sdk-core/stream-text)_

### 6. Agentic AI 在创意写作中的崛起

**2026 年最重大的技术趋势**：从"工具"到"智能代理"。

**创意写作 Agent 架构模式**：
- **GOAT-Storytelling-Agent**（开源）：多阶段规划+写作流水线，从上至下构建故事。仅需输入一个话题即可自主生成整部中篇小说
- **角色化框架**：灵感缪斯（头脑风暴）+ 情节建筑师（结构）+ 世界构建者（一致性）+ 散文风格师（润色）——四种 AI 伙伴角色
- **多 Agent 协作**：多个 AI Agent 传递上下文、共享长期记忆、协调决策

**Agent 的四要素**：
1. 目标导向推理（将高级目标分解为子任务）
2. 多步执行（规划→执行→自我纠正）
3. 记忆系统（短期上下文 + 长期向量存储）
4. 工具使用（文件系统、API、代码执行）

**对 WriteTeam 的意义**："隐形编辑"愿景的技术实现路径——不是单个 API 调用，而是一个理解你整部小说、能自主分析和建议的 Agent 系统。可以先从"统一搭档对话"开始，将 22 个 AI 工具统一为一个理解上下文的对话式 Agent。

_Source: [GOAT-AI-lab](https://github.com/GOAT-AI-lab/GOAT-Storytelling-Agent), [Kellton](https://www.kellton.com/kellton-tech-blog/generative-ai-2-0-agentic-workflows-2026), [Data Science Dojo](https://datasciencedojo.com/blog/agentic-llm-in-2025/)_

1. **"故事意识引擎"= Context 工程 + RAG + Agentic AI 的融合**——这不是三个独立功能，而是一个统一的架构
2. **"隐形编辑"= 分析型 AI 嵌入生成式平台**——将 Marlowe 的分析能力融入写作过程而非作为独立报告
3. **Copilot 式内联续写 = 市场空白中的蓝海**——目前无任何小说写作平台实现
4. **中文原生体验 = 无竞争的细分市场**——所有竞品均为英文优先

### 风险与挑战

| 风险 | 应对策略 |
|------|---------|
| 模型快速迭代导致技术栈频繁变动 | BYOK + 模型无关架构，解耦业务逻辑与具体模型 |
| 上下文窗口成本随文本量线性增长 | 智能上下文精选 + Prompt Caching + 摘要分层 |
| AI 散文质量尚不稳定 | 定位为"辅助"而非"替代"，强调人机协作 |
| 中文 LLM 创意写作能力相对较弱 | 利用 Prose Mode + 风格样本弥补；长期推动微调 |
| RAG 实现复杂度高 | 渐进式实施：先优化静态上下文注入 → 再引入向量检索 |

_Source: [Lakera](https://www.lakera.ai/blog/prompt-engineering-guide), [Novelcrafter Blog](https://www.novelcrafter.com/blog/fine-tuning-ai-for-authors), [Procedure.tech](https://procedure.tech/blogs/the-streaming-backbone-of-llms-why-server-sent-events-(sse)-still-wins-in-2025)_

## 研究综合与战略建议

### 执行摘要

2026 年 AI 辅助小说写作工具市场正经历从"文本生成器"到"写作操作系统"的范式转变。市场规模约 19-35 亿美元（CAGR 23-27%），中国以 67% 采用率全球领先，但**所有主流产品均为英文优先**，中文创意写作是一个完全空白的细分市场。

**核心洞察：** 竞争的本质已从"谁的 AI 写得更好"转向"谁的写作工作流更智能"——知识库深度（Story Bible/Codex）、上下文工程、风格保留、分析能力构成了真正的护城河。

WriteTeam 已具备坚实的技术基础（BYOK + SSE 流式 + Story Bible + Saliency + 22 个 AI 工具），且头脑风暴确定的"隐形编辑"愿景精准命中了多个未被满足的市场需求。

### WriteTeam 战略定位

基于全面研究，WriteTeam 应定位为：

> **中文原生的"隐形编辑"写作平台** — 融合 Sudowrite 的创意灵感能力、Novelcrafter 的知识库深度、和 Marlowe 的分析智慧，为中文小说作者打造"读过你全书的隐形编辑"。

**与竞品的差异化定位：**

```
WriteTeam 不是任何竞品的中文翻版。
WriteTeam = Sudowrite 的创意引擎
           + Novelcrafter 的知识库深度
           + Marlowe 的分析智慧
           + 无人做过的内联续写体验
           + 中文原生
```

### 五大战略机会（按优先级排序）

#### 机会 1：故事意识引擎（P0 — 核心差异化）

**是什么**：统一 Story Bible + Saliency + RAG 为"理解全书的上下文层"
**为什么**：上下文管理是 2026 年的核心竞争力。Novelcrafter 的 Codex 已证明结构化知识库 + 智能注入的巨大价值。"先上下文，再散文"是行业共识最佳实践。
**怎么做**：
- 将 Story Bible 条目向量化，按场景需求智能检索
- 实施多层记忆：全书压缩摘要 → 章节摘要 → 当前场景详情
- 参考 Novelcrafter 的 Scene Summaries 机制——前序场景摘要自动注入后续 Prompt
- 参考 Novelcrafter 的 Progressions——记录角色随时间变化（非静态笔记）

#### 机会 2：Copilot 式内联续写（P1 — 市场蓝海）

**是什么**：编辑器中真正的灰色幽灵文字续写，Tab 键接受
**为什么**：**截至 2026 年无任何主流小说写作工具实现**。Sudowrite 的 Write 最接近，但结果在侧栏而非内联。这是确认的市场空白。
**怎么做**：
- 基于 TipTap 编辑器实现内联灰色文字渲染
- 利用 SSE 流式逐 token 显示续写建议
- 结合故事意识引擎注入上下文，确保续写贴合角色/情节/风格

#### 机会 3：统一搭档对话（P1 — 交互革新）

**是什么**：将 22 个独立 AI API routes 统一为一个上下文感知的对话 Agent
**为什么**：Sudowrite 有 Describe/Expand/Brainstorm/Rewrite 等独立工具，但用户需要手动选择。2026 年 Agentic AI 趋势表明：自然语言入口 + AI 自动理解意图并路由到正确功能是更优体验。
**怎么做**：
- 构建统一对话 Agent，自然语言触发不同写作功能
- Agent 自动基于上下文判断是续写/改写/扩写/缩写/头脑风暴
- 保留直接工具入口供偏好精确控制的用户

#### 机会 4：深度编辑智慧（P2 — 分析 + 生成融合）

**是什么**：将 Marlowe 级别的手稿分析能力嵌入生成式写作平台
**为什么**：Marlowe 只分析不生成，Sudowrite/Novelcrafter 只生成不深度分析。融合两者是未被实现的需求，且精准对应"隐形编辑"愿景。
**怎么做**：
- **节奏心电图**：分析段落长度/对话密度/动作节奏变化，可视化展示
- **角色弧线检测**：追踪角色情感/关系/能力变化曲线
- **一致性检查增强**：超越当前简单检查，实现跨章节的角色特征/世界观规则一致性验证
- **模拟读者视角**：分析哪些段落可能让读者困惑/无聊/兴奋

#### 机会 5：作者声音保留（P1 — 信任构建）

**是什么**：中文版的"My Voice"——用作者写作样本动态调整 AI 输出风格
**为什么**：82% 作者担忧 AI 同质化风格（Authors Guild 2024）。Sudowrite 的 My Voice 仅支持英文。中文作者的风格保留需求完全空白。
**怎么做**：
- 短期：增加"作者风格样本"功能，将样本作为 few-shot 示例注入 Prompt
- 中期：Prose Mode 扩展为可自定义风格模板
- 长期：探索中文创意写作微调模型

### 实施路线图

```
Phase 1（0-3个月）: 故事意识引擎 MVP
├── 多层记忆架构（全书摘要 + 章节摘要 + 场景详情）
├── Scene Summaries 自动注入后续 Prompt
├── 角色 Progressions（动态变化记录）
└── 优化 story-context.ts 上下文编排

Phase 2（3-6个月）: 交互革新
├── Copilot 式内联续写（幽灵文字 + Tab 接受）
├── 统一搭档对话 Agent（合并 22 个 AI 工具）
├── 作者风格样本功能
└── Smart Highlighting（过度使用词/AI模式检测）

Phase 3（6-12个月）: 深度编辑智慧
├── 节奏心电图可视化
├── 角色弧线检测与追踪
├── 跨章节一致性验证
├── 模拟读者视角分析
└── RAG 向量化知识库（Story Bible → 向量存储）

Phase 4（12+个月）: 生态扩展
├── 多模型堆栈优化（按任务自动推荐模型）
├── 中文创意写作微调模型探索
├── 导出增强（排版/分发就绪）
└── 协作功能
```

### 从竞品学到的最重要教训

1. **Sudowrite 教会我们**：风格匹配（My Voice）和专业写作模型（Muse）的巨大价值——通用 AI 的散文质量无法与专门微调的模型竞争
2. **Novelcrafter 教会我们**：知识库深度（Codex）和 Prompt 透明化是长篇小说一致性的基石——用户应能看到发送给 AI 的精确 Prompt
3. **Marlowe 教会我们**：分析型 AI 与作者之间有天然信任——"不写文字只分析"消除了 AI 替代作者的恐惧，建立"编辑顾问"定位
4. **DreamGen 教会我们**：Steering Mode（实时调节 AI 行为）是用户控制感的关键——作者需要觉得自己在掌控 AI，而非被 AI 掌控
5. **行业共识教会我们**："先投入上下文，再生成散文"是第一法则。花 30 分钟建 Story Bible 可以省下数小时的一致性修复
