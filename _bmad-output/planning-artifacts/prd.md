# Product Requirements Document - writeteam

**Author:** Elij
**Date:** 2026-03-01

## Executive Summary

WriteTeam 是面向中文小说作者的 AI 创作搭档平台。当前版本（v1）已实现完整的写作基础设施——BYOK AI 配置、Story Bible 上下文系统、22 个 AI 写作工具、Canvas 可视化规划、剧集管理、导入/导出。本 PRD 定义 v2 阶段的核心演进方向：**将 WriteTeam 从"写作工具集合"升级为"读过你全书的隐形编辑"。**

**目标用户：** 中文长篇小说作者（网文作者、类型小说作者、严肃文学作者），核心痛点为：长篇连续性崩溃（30万字后记不住细节）、AI 生成缺乏个人风格、无法获得深度编辑反馈、写作中频繁的上下文切换打断心流。

**核心问题：** 2026 年 AI 写作工具市场（~19-35 亿美元，CAGR 23-27%）正从"文本生成器"演进为"写作操作系统"。Sudowrite 领先于创意灵感，Novelcrafter 领先于知识库深度，Marlowe 领先于手稿分析——但无一产品融合三者，且**所有主流产品均为英文优先**。中文创意写作是一个完全空白的细分市场，中国同时拥有全球最高的 AI 写作采用率（67%）。

**核心愿景：** 隐形编辑——安静在旁边，你需要时它就在，越用越聪明。不是你要去"使用"的工具，而是一个逐渐懂你的存在。它有大脑（故事意识引擎）、有直觉（幽灵文字续写）、有心（创作见证系统）、有眼睛（多维度分析视图）、会说话（统一搭档对话）。

### What Makes This Special

**差异化定位：** WriteTeam 不是任何竞品的中文翻版。它是 Sudowrite 的创意引擎 + Novelcrafter 的知识库深度 + Marlowe 的分析智慧 + 无人做过的内联续写体验 + 中文原生。

**五大独占优势：**

1. **故事意识引擎** — 统一 Story Bible + Saliency + 多层记忆为"理解全书的上下文层"。参考 Novelcrafter Codex 的结构化知识库 + 智能注入，但更进一步：自动摘要、动态角色追踪（Progressions）、场景摘要自动传递。这是"先上下文，再散文"这一 2026 年行业第一法则的产品化实现。

2. **Copilot 式内联续写（幽灵文字）** — 截至 2026 年，**无任何主流小说写作工具实现真正的编辑器内联灰色续写文本**。Sudowrite 最接近但结果在侧栏。这是经领域研究确认的市场蓝海。

3. **分析+生成融合** — Marlowe 只分析不生成，Sudowrite/Novelcrafter 只生成不深度分析。WriteTeam 将节奏心电图、角色弧线检测、一致性验证、模拟读者视角嵌入写作流程——分析型 AI 与生成型 AI 的首次融合。

4. **统一搭档对话** — 将 22 个独立 AI 工具统一为一个上下文感知的对话 Agent。自然语言入口替代功能菜单，AI 自动理解写作意图并路由到正确功能。对应 2026 年 Agentic AI 趋势。

5. **中文原生体验** — 中文小说的独特需求（人称系统、修辞传统、网文/传统文学双轨、中文韵律分析）无任何竞品覆盖。中文优先不只是翻译 UI，是理解中文创作的灵魂。

## Success Criteria

### User Success

**核心体验指标——"隐形编辑"感知**

| 指标 | 定义 | 目标 |
|------|------|------|
| **故事理解感知** | 作者认为 AI "理解我的故事"的比例（基于 AI 生成内容与前文一致性的用户评分） | ≥70% 的 AI 交互被评为"贴合上下文" |
| **续写接受率** | 幽灵文字续写被 Tab 接受的比例 | ≥25%（参考 GitHub Copilot 的 ~30% 接受率） |
| **心流连续性** | 单次写作 session 中用户不因工具操作中断写作的平均时长 | 从 v1 基线提升 ≥40% |
| **回归仪式触达** | 打开项目时"写作仪式感"引导的用户参与率 | ≥60% 用户与回归引导互动 |
| **搭档对话使用** | 统一搭档对话替代直接工具按钮的比例 | ≥50% 的 AI 交互通过自然语言对话发起 |

**连续性与一致性指标**

| 指标 | 定义 | 目标 |
|------|------|------|
| **跨章节一致性** | 5 万字以上项目中 AI 生成内容出现角色/设定矛盾的频率 | 较 v1 降低 ≥60% |
| **上下文覆盖率** | Story Bible 条目被 AI 正确引用/遵守的比例 | ≥85% |
| **长篇可用性** | 10 万字+ 项目中 AI 工具的响应质量保持率 | 质量衰减 ≤15%（vs 大上下文窗口的典型 40-50% 衰减） |

### Business Success

| 时间 | 指标 | 目标 |
|------|------|------|
| **3 个月** | 活跃用户（周活） | 从 v1 基线增长 ≥3x |
| **3 个月** | 新用户 7 日留存率 | ≥40% |
| **3 个月** | 核心功能（故事意识引擎 + 幽灵文字）日活使用率 | ≥60% 的活跃用户每日触达 |
| **6 个月** | 用户项目平均字数 | ≥3 万字（证明长篇场景可用） |
| **12 个月** | 月活留存率 | ≥30%（创作工具行业优秀水平） |
| **12 个月** | 用户 NPS（净推荐值） | ≥50 |
| **12 个月** | Story Bible/Codex 条目数 | 活跃项目平均 ≥20 条（证明知识库深度被使用） |

### Technical Success

| 指标 | 目标 | 说明 |
|------|------|------|
| **AI 响应延迟（首 token）** | ≤1.5s（P95） | SSE 流式首 token 时间，影响幽灵文字体验 |
| **上下文组装时间** | ≤500ms | story-context.ts 多层记忆 + RAG 检索的总耗时 |
| **编辑器输入延迟** | ≤50ms | TipTap 编辑器在幽灵文字插件激活时的输入响应 |
| **自动摘要准确率** | ≥80% | 自动生成的章节摘要被作者确认无需修改的比例 |
| **多模型兼容** | ≥5 provider | BYOK 模式下故事意识引擎在不同模型上的功能完整性 |
| **数据安全** | 零泄露 | 用户手稿和 API Key 的安全性——RLS + AES-256-GCM |
| **构建/部署** | 零回归 | v2 功能不破坏 v1 已有功能 |

### Measurable Outcomes

**v2 的"北极星指标"：每周活跃写作字数**——衡量 WriteTeam 是否真正帮助作者写得更多、写得更好。

**关键里程碑：**
1. **Phase 1 完成**（故事意识引擎 MVP）：10 万字+ 项目的 AI 一致性较 v1 提升 ≥60%
2. **Phase 2 完成**（幽灵文字 + 统一对话）：用户平均写作 session 时长提升 ≥30%
3. **Phase 3 完成**（深度编辑智慧）：≥50% 活跃用户使用分析功能（节奏/角色弧/一致性）

## Product Scope

v2 采用三阶段递进开发，故事意识引擎为一切后续功能的地基。详细分阶段能力清单、依赖关系和风险缓解见下方 "Project Scoping & Phased Development" 章节。

| 阶段 | 核心主题 | 关键能力 |
|------|---------|---------|
| **Phase 1（MVP）** | 故事意识引擎 | 多层记忆、Scene Summaries 自动生成、角色 Progressions、差异化上下文组装、写作仪式感 |
| **Phase 2** | 交互革新 | 幽灵文字、统一搭档对话、作者风格样本、Smart Highlighting |
| **Phase 3** | 深度编辑智慧 | 节奏心电图、角色弧线检测、一致性验证、模拟读者、完稿体检、RAG 向量化 |
| **远期** | 愿景扩展 | 多模型堆栈优化、角色对话模拟器、分支宇宙、创作见证系统、移动端灵感捕手、协作功能 |

**MVP 验证标准：** 5 万字以上项目中，AI 生成内容的角色/设定一致性较 v1 显著提升，用户无需频繁手动纠正。

## User Journeys

### Journey 1：陈默（网文日更作者）— 核心成功路径

**人物：** 陈默，28 岁，全职网文作者，正在连载一部 120 万字的都市玄幻小说《灵气复苏》。每天需要更新 6000 字，已写到第 380 章。最大的痛苦是记不清前面埋的伏笔和角色设定——上周读者在评论区指出他把配角的左右手写反了。

**Opening Scene：**
陈默打开 WriteTeam，今天要写第 381 章。编辑器显示一行温柔的引导："上次你停在林澈进入地下遗迹的入口。遗迹守卫提到了'玉佩之主'——这条线索连接着第 47 章和第 213 章的伏笔。" 陈默愣了一下——他自己都快忘了玉佩的事了。

**Rising Action：**
陈默开始写。光标停顿了两秒，编辑器中浮现灰色的幽灵文字："林澈握紧腰间的玉佩，感受到一股温热从掌心涌入——和当初在秦家密室中第一次触碰时一模一样。" 陈默觉得方向不错但措辞要改，他 Tab 接受了前半句，后半句自己重写。继续写对话时，他直接在编辑器里打字："帮我写一段林澈和遗迹守卫的对白，守卫应该知道玉佩的来历但不愿直说。" 统一搭档对话理解了意图，基于 Story Bible 中守卫的性格档案和玉佩的线索链，生成了三个版本的对白。

**Climax：**
写到 4000 字时，侧栏弹出一条安静的提示："提醒：你在第 213 章设定了玉佩只有在满月时才会发热。当前章节的时间线是农历十二，非满月。" 陈默倒吸一口气——差点又写出一个设定矛盾。他修改了描写，将"温热"改为"微微震颤"。

**Resolution：**
6000 字完成。陈默在 90 分钟内完成了日更，比之前平均节省了 30 分钟的"翻回去找细节"时间。他不再担心设定崩塌，因为隐形编辑在默默守护。

**揭示的能力需求：**
- 故事意识引擎：全书上下文感知、伏笔追踪、时间线一致性
- 幽灵文字：内联续写 + Tab 接受
- 统一搭档对话：自然语言触发 AI 写作功能
- 写作仪式感：项目回归引导
- 连续性守护：实时矛盾检测与提醒

### Journey 2：林小雨（类型小说新手作者）— 冷启动 + 学习路径

**人物：** 林小雨，22 岁，大学毕业生，从小爱读悬疑推理小说，决定自己写一部。她有一个模糊的构思——一桩发生在民国上海的密室杀人案——但从未写过完整的小说，不知道如何组织 10 万字以上的叙事。

**Opening Scene：**
林小雨注册 WriteTeam，创建第一个项目"沪上迷局"。她在设置中配置了 DeepSeek 的 API Key（因为便宜），选择了"悬疑推理"类型。面对空白页，她打开了搭档对话，打字："我想写一个民国上海的密室杀人案，但不知道怎么开始。"

**Rising Action：**
搭档对话引导她一步步构建 Story Bible：主角身份（租界巡捕房探长）、密室设定（法租界一栋洋楼的书房）、嫌疑人列表、核心诡计。每当她描述一个角色，系统自动创建 Story Bible 条目并提问："这个角色有什么秘密？" 30 分钟后，她有了 8 个 Story Bible 条目和一个粗略大纲。

她开始写第一章。写了 500 字后犯难——不确定是用第一人称还是第三人称。她在搭档对话中问："悬疑小说用什么人称好？" AI 基于类型知识给出建议，并提供了两种人称的 200 字对比样本。她选了限制性第三人称。

**Climax：**
写到第三章时，林小雨发现自己的诡计有逻辑漏洞——密室的窗户在第一章写的是"被钉死的"，但第三章的解答需要凶手从窗户逃离。系统在她写到窗户相关描写时弹出提示："第 1 章第 3 段设定：书房窗户从内部钉死，无法从外部打开。当前描写似乎与此矛盾。" 她意识到需要修改诡计设计，而不是写出一个读者能一眼看穿的 bug。

**Resolution：**
一个月后，林小雨完成了 8 万字的初稿。她从未写过这么长的东西。Story Bible 从最初 8 条增长到 35 条（许多是系统自动提取并建议她确认的）。她用"完稿体检"功能检查了节奏分布——发现中段有一段 15000 字的节奏过于平缓，根据建议进行了修改。

**揭示的能力需求：**
- 冷启动引导：从零构建 Story Bible 的对话式工作流
- BYOK 配置：简单直观的 API Key 设置
- 类型感知：悬疑推理类型的特化建议
- 人称/时态控制：POV 设置与一致性保持
- 自动实体提取：AI 自动建议创建 Story Bible 条目
- 完稿体检：节奏分析 + 改进建议

### Journey 3：周远航（严肃文学作者）— 边缘场景 + 风格保留

**人物：** 周远航，45 岁，省作协会员，已出版两部纯文学长篇。正在写第三部关于三线城市拆迁的现实主义小说。他对 AI 持怀疑态度——担心 AI 会让他的文字变得"塑料味"。他试用 WriteTeam 是因为一个作家朋友推荐。

**Opening Scene：**
周远航创建项目后，第一件事是上传了自己前两部小说中最满意的三个章节（约 15000 字）作为风格样本。系统分析后展示了他的风格画像：长句为主、内心独白丰富、善用环境隐喻、对话克制简短、叙事节奏偏慢。他看了看，觉得还挺准的。

**Rising Action：**
周远航不用幽灵文字——他把续写功能关了，他只想自己写。但他频繁使用搭档对话来讨论情节走向。他问："主角张建国在得知拆迁补偿被克扣后，应该是隐忍还是爆发？" AI 的回答不是给出"正确答案"，而是从两个角度分析了叙事效果——隐忍更符合前三章建立的角色性格，但爆发能制造转折。最终它说："基于你的风格样本，你的角色通常在沉默中积蓄力量——隐忍可能更'你'。"

**Climax：**
写到 12 万字时，周远航请求"完稿体检"。报告显示他的角色弧线分析——张建国从沉默到觉醒的弧线清晰，但配角刘芳的情感线在第 8 章之后几乎消失了。报告还指出他的段落长度均值比前两部作品偏短了 15%，风格可能在"漂移"。他重新检查了中段，发现确实在赶进度时风格有所松懈。

**Resolution：**
周远航对 WriteTeam 的态度从怀疑变为信任——不是因为 AI 能写他的文字，而是因为 AI 能像一个读过他全部作品的编辑那样，告诉他"这段不像你"。他没有用任何生成功能，但故事意识引擎和分析功能成了他的核心工具。

**揭示的能力需求：**
- 作者风格样本：上传写作样本 → 风格画像分析
- 可关闭的生成功能：尊重不需要 AI 写作的用户
- 深度对话：搭档对话基于角色性格和作者风格给出分析式建议（非生成式）
- 角色弧线分析：追踪角色情感变化曲线，检测配角消失
- 风格漂移检测：与作者自身基线对比
- 完稿体检报告：综合分析（节奏/角色/风格/结构）

### Journey 4：系统自助管理 — AI 配置与数据管理

**人物：** 任何 WriteTeam 用户（以陈默为例）在非写作场景下的管理操作。

**Opening Scene：**
陈默最近从 DeepSeek 换到了 OpenRouter，因为想试试 Claude Opus 写对话场景的效果。他进入设置页面，添加 OpenRouter 的 API Key 和 Base URL，选择模型。

**Rising Action：**
他想为不同场景配置不同模型：续写用 DeepSeek（便宜），角色对话用 Claude Opus（质量高），一致性检查用 GPT-4.1（逻辑强）。系统支持按功能类型推荐和切换模型。

他还需要管理数据：导出《灵气复苏》的完整项目（含 Story Bible 和全部章节）备份到本地。他使用 docx 导出功能，同时导出 Story Bible 为 JSON 格式。

**Climax：**
一次误操作中，陈默不小心删除了一个关键的 Story Bible 条目。系统提供了"最近删除"的恢复窗口（7 天内可恢复），他成功找回了数据。

**Resolution：**
陈默的 AI 配置和数据管理完全自助完成，无需联系任何人。BYOK 模式让他对自己的数据和成本有完全控制权。

**揭示的能力需求：**
- 多模型配置：按功能类型指定不同模型
- 数据导出：项目完整导出（docx + Story Bible JSON）
- 数据安全恢复：误删恢复机制
- BYOK 自助管理：API Key 管理、用量监控

## Innovation & Novel Patterns

### Detected Innovation Areas

**1. Copilot 式内联续写（幽灵文字）— 行业首创**

截至 2026 年，无任何主流小说写作工具实现真正的编辑器内联灰色续写文本。Sudowrite 最接近但结果在侧栏显示，Novelcrafter 使用 Scene Beats 但需手动触发，NovelAI 使用独立文本框。WriteTeam 将 GitHub Copilot 的"灰色幽灵文字 + Tab 接受"范式首次引入创意写作领域，通过 TipTap Suggestion 插件 + SSE 流式渲染 + 编辑器装饰层实现。这不是渐进式改进，而是交互范式的根本改变——续写从"主动调用工具"变成"呼吸般自然"。

**2. 分析 + 生成融合 — 品类融合创新**

当前市场存在清晰的分野：Marlowe/Authors.ai 只做手稿分析（节奏、角色弧、一致性），Sudowrite/Novelcrafter 只做内容生成。WriteTeam 首次将节奏心电图、角色弧线检测、一致性验证、模拟读者视角等分析能力嵌入生成式写作流程。分析不是事后检查，而是写作过程中的实时守护——写到矛盾处即时提醒，而非完稿后才发现。

**3. 故事意识引擎 — 上下文工程创新**

受 Andrej Karpathy "LLM is CPU, context window is RAM, you are the OS" 思想启发，故事意识引擎将 Story Bible 结构化知识 + 客户端 Saliency 分析 + 多层记忆（全书压缩摘要→章节摘要→当前场景详情）统一为自动维护的"故事理解层"。对比 Novelcrafter Codex（需手动维护）和 Sudowrite（无持久记忆），这是首个自动摘要 + 动态角色追踪（Progressions）+ 按功能类型差异化上下文组装的系统。核心创新在于：用智能压缩让任何 BYOK 模型都能"记住"百万字长篇，解决大上下文窗口 40-50% 质量衰减问题。

**4. 统一搭档对话 — Agentic AI 在创意写作的落地**

将 22 个独立 AI 工具统一为一个上下文感知的对话 Agent，是 2026 年 Agentic AI 趋势在创意写作领域的首次产品化。用户不再需要记住"续写用哪个按钮、改写用哪个按钮"，而是用自然语言描述意图，AI 自动理解并路由到正确功能。这参考了 GOAT-Storytelling-Agent 的多 Agent 协作架构，但更进一步——从"工具菜单"到"对话伙伴"的范式转换。

**5. 中文原生创意写作 — 市场空白创新**

全球 AI 写作工具市场的主流产品（Sudowrite、Novelcrafter、NovelAI、DreamGen）均为英文优先。中国拥有全球最高的 AI 写作采用率（67%），但中文创意写作是一个完全空白的细分市场。WriteTeam 的中文原生不只是 UI 翻译，而是深入中文创作的灵魂：人称系统（你/他/她 vs I/he/she）、修辞传统（四字格、平仄、叠词）、网文/传统文学双轨体系（3000 字爽点 vs 意象密度）、中文韵律分析。

## Web Application Specific Requirements

### Project-Type Overview

WriteTeam 是基于 Next.js 16 App Router 的单页应用（SPA），采用客户端渲染为主 + 服务端路由守护的混合架构。核心编辑器场景（TipTap 富文本 + AI 流式交互 + Canvas 可视化）要求高度的客户端交互能力，而 Dashboard/Settings 等管理页面利用 Server Components 优化首屏加载。

### Technical Architecture Considerations

**应用架构模式：SPA + SSR 混合**

| 路由组 | 渲染策略 | 说明 |
|--------|---------|------|
| `(auth)/` | Server Components | 登录/注册，SEO 不敏感，服务端渲染 |
| `(dashboard)/` | Server Components + Client Islands | 项目列表/设置，Server Components 首屏 + 交互部分客户端水合 |
| `(editor)/` | 纯客户端渲染 | TipTap 编辑器 + AI 工具 + 幽灵文字，完全客户端交互 |
| `(editor)/canvas/` | 纯客户端渲染 | @xyflow/react Canvas，纯客户端 |
| `api/ai/*` | Route Handlers | 21+ AI 端点，SSE 流式响应 |

**实时交互需求：**
- **AI 流式输出**：SSE（Server-Sent Events）协议，`createOpenAIStreamResponse()` 处理 OpenAI-compatible 流式响应
- **幽灵文字（v2 新增）**：SSE 流式 + TipTap Decoration 实时渲染灰色续写文本，首 token 延迟 ≤1.5s
- **自动保存**：1s debounce 保存到 Supabase，无需 WebSocket（非协作场景）
- **Saliency 分析**：客户端 5s debounce 启发式分析，不依赖服务端

## Project Scoping & Phased Development

### MVP Strategy & Philosophy

**MVP 方式：体验型 MVP（Experience MVP）**

WriteTeam v2 不是从零开始，而是在已完成的 v1（6 Epic / 22 Story / 10 DB migrations）基础上增量构建。MVP 策略不是"最小功能集"，而是"最小体验转变"——用最少的新功能让用户感知到从"工具集合"到"隐形编辑"的质变。

**MVP 核心判断标准：** 如果用户在 v2 写了 5 万字后仍然觉得和 v1 没有本质区别，MVP 就失败了。反过来，如果用户第一次体验到"AI 记住了我上一章写的内容"，MVP 就成功了。

**资源假设：** 单人全栈开发（Elij），利用 v1 已有基础设施，预计 MVP 周期 8-12 周。

### MVP Feature Set (Phase 1) — 故事意识引擎核心

**核心用户旅程支撑：** 陈默（日更网文）的核心成功路径

**Must-Have 能力清单：**

| # | 能力 | 说明 | 验证标准 |
|---|------|------|---------|
| 1 | **多层记忆架构** | 全书压缩摘要 + 章节摘要 + 当前场景详情，替代当前 story-context.ts 的扁平上下文 | 5 万字项目 AI 一致性提升 ≥60% |
| 2 | **Scene Summaries 自动生成** | 场景/章节完成后自动生成摘要，注入后续 AI Prompt | 自动摘要准确率 ≥80% |
| 3 | **角色 Progressions** | 角色动态变化记录（非静态 Story Bible 条目），追踪角色状态随章节演变 | Story Bible 条目被 AI 正确引用 ≥85% |
| 4 | **差异化上下文组装** | 按功能类型（写作/规划/检查）组装不同的上下文内容，优化 token 使用 | 上下文组装时间 ≤500ms |

**为什么这样切：** Phase 1 聚焦"大脑"——让 AI 真正理解故事。没有故事理解能力，幽灵文字只会生成与前文矛盾的内容，统一对话也只是套了壳的工具菜单。故事意识引擎是一切后续功能的地基。

| 风险 | 缓解 |
|------|------|
| 用户不愿意配置 BYOK（嫌麻烦） | 1) 极简配置引导（3 步完成）；2) 推荐最便宜的 provider（DeepSeek）；3) 未来考虑免费 Tier |
| 中文 AI 写作市场被巨头（字节/百度）快速占领 | 1) 聚焦长篇深度场景（巨头通常做通用场景）；2) 社区驱动 vs 资本驱动；3) BYOK 灵活性是差异化 |
| 用户觉得 AI 生成内容"不像自己写的" | 1) Phase 2 风格样本功能；2) 所有 AI 输出可编辑/拒绝；3) 强调 AI 是"建议"不是"替代" |
| 中文市场对 AI 写作的版权/监管风险 | BYOK 模式将 AI 调用责任转移给用户；遵守《生成式AI管理办法》；不存储生成内容在服务器端 |
| 自动摘要/实体提取准确率不足致用户信任崩塌 | 所有自动生成内容标记为"建议"，用户可审核修正；降级方案：保持手动 Story Bible |

**资源风险：**

| 风险 | 缓解 |
|------|------|
| 单人开发进度瓶颈 | 1) 严格 Phase 分期，不跨阶段并行；2) 优先利用 v1 已有基础设施；3) AI 辅助开发加速 |
| 技术债务积累 | 1) 每个 Phase 结束做回顾；2) v2 架构设计预留扩展点；3) 保持 v1 代码质量标准 |

## Functional Requirements

### 故事意识与上下文管理

- **FR1:** 系统可为每个项目自动维护多层记忆结构（全书压缩摘要、章节摘要、当前场景详情） `Phase 1`
- **FR2:** 系统可在章节/场景完成后自动生成该章节的结构化摘要 `Phase 1`
- **FR3:** 作者可审核、编辑或拒绝系统自动生成的章节摘要 `Phase 1`
- **FR4:** 系统可为每个角色维护动态变化记录（Progressions），追踪角色状态随章节的演变 `Phase 1`
- **FR5:** 系统可按 AI 功能类型（写作/规划/检查）差异化组装上下文内容 `Phase 1`
- **FR6:** 系统可从作者写作内容中自动提取实体（角色、地点、物品）并建议创建 Story Bible 条目 `Phase 1`
- **FR7:** 系统可在 AI 生成内容时自动注入相关的章节摘要和角色 Progressions `Phase 1`
- **FR8:** 系统可对 Story Bible 条目进行向量化存储，支持语义检索 `Phase 3`
- **FR9:** 系统可追踪故事时间线，在检测到时间矛盾时提醒作者 `Phase 3`

### AI 写作辅助

- **FR10:** 作者可在编辑器中获得 Copilot 式内联灰色续写文本（幽灵文字），通过 Tab 接受或 Esc 忽略 `Phase 2`
- **FR11:** 系统可实时流式渲染幽灵文字，确保续写文本基于故事意识引擎的上下文 `Phase 2`
- **FR12:** 作者可开启或关闭幽灵文字功能 `Phase 2`
- **FR13:** 作者可使用全部已有 AI 写作工具（续写、改写、扩写、缩写、头脑风暴等 22 个工具） `v1 保留`
- **FR14:** 作者可通过选中文本触发浮动 AI 菜单进行文本操作 `v1 保留`
- **FR15:** 作者可上传写作样本，系统可分析并生成作者风格画像 `Phase 2`
- **FR16:** 系统可使 AI 生成内容自动匹配作者风格画像 `Phase 2`
- **FR17:** 系统可检测并标记文本中过度使用的词汇和 AI 生成模式 `Phase 2`
- **FR18:** 系统可根据作者选择的小说类型（如悬疑、玄幻、言情）提供类型特化的 AI 建议和反馈 `Phase 2`
- **FR19:** 作者可为项目设置叙事人称（第一人称/第三人称限制/全知），系统可确保 AI 生成内容遵守人称设定 `Phase 2`

### 统一搭档交互

- **FR20:** 作者可通过自然语言对话描述写作意图，系统自动理解并路由到对应 AI 功能 `Phase 2`
- **FR21:** 统一搭档对话可感知当前写作上下文（打开的文档、选中的文本、Story Bible 状态） `Phase 2`
- **FR22:** 系统可在作者打开项目时展示上下文回归引导（上次写作位置、相关伏笔、未完成线索） `Phase 1`
- **FR23:** 作者可在搭档对话中获得基于角色性格和作者风格的分析式建议（非仅生成式） `Phase 2`

### 深度编辑与分析

- **FR24:** 系统可生成叙事节奏可视化（段落长度、对话密度、动作节奏分布图） `Phase 3`
- **FR25:** 系统可追踪角色情感变化弧线，检测角色弧线停滞或配角消失 `Phase 3`
- **FR26:** 系统可执行跨章节一致性验证，检测角色/设定/时间线矛盾 `Phase 3`
- **FR27:** 系统可在作者写作过程中检测到角色属性、时间线或地点设定的矛盾时，通过侧栏非阻塞通知提醒作者 `Phase 3`
- **FR28:** 系统可模拟第一次读者视角，逐章生成结构化读者反应报告（包含悬念感知、情感节奏、困惑点） `Phase 3`
- **FR29:** 系统可生成完稿体检报告，综合诊断角色、节奏、风格、结构 `Phase 3`
- **FR30:** 系统可检测作者风格漂移（与自身基线或风格样本对比） `Phase 3`

### Story Bible 与知识管理

- **FR31:** 作者可创建、编辑、删除 Story Bible 条目（角色、地点、物品、设定） `v1 保留`
- **FR32:** 作者可为 Story Bible 条目设置可见性（全局/项目级/私有） `v1 保留`
- **FR33:** 系统可在 AI 交互中自动引用相关的 Story Bible 条目 `v1 增强`
- **FR34:** 作者可查看 AI 自动学到的故事知识面板，审核并修正 AI 的理解 `Phase 2`
- **FR35:** 系统可在误删 Story Bible 条目后提供恢复窗口 `Phase 1`

### 项目与文档管理

- **FR36:** 作者可创建、管理项目和剧集（系列） `v1 保留`
- **FR37:** 作者可在项目内创建、编辑、排序、删除文档 `v1 保留`
- **FR38:** 系统可自动保存文档内容，维护纯文本镜像和字数统计 `v1 保留`
- **FR39:** 作者可使用 Canvas 进行可视化故事规划 `v1 保留`
- **FR40:** 作者可导入 docx 文件和导出项目为 docx 格式 `v1 保留`
- **FR41:** 作者可导出 Story Bible 为 JSON 格式 `Phase 1`

### AI 配置与多模型管理

- **FR42:** 作者可配置自己的 LLM API Key 和 Base URL（BYOK 模式） `v1 保留`
- **FR43:** 作者可从预设 Provider 列表选择配置（DeepSeek、OpenAI、Ollama、OpenRouter、硅基流动） `v1 保留`
- **FR44:** 作者可为不同 AI 功能类型指定不同的模型 `Phase 2`
- **FR45:** 系统可按功能类型推荐适合的模型 `Phase 2`
- **FR46:** 系统可安全加密存储用户的 API Key `v1 保留`

### 用户账号与安全

- **FR47:** 用户可注册、登录、登出账号 `v1 保留`
- **FR48:** 系统可通过 Row Level Security 确保用户只能访问自己的数据 `v1 保留`
- **FR49:** 系统可在每次请求时刷新认证会话 `v1 保留`

## Non-Functional Requirements

### Performance

WriteTeam 的性能直接影响写作心流。编辑器输入延迟和 AI 响应速度是产品体验的生命线。

| NFR# | 需求 | 指标 | 说明 |
|------|------|------|------|
| **NFR1** | 编辑器输入响应不因 AI 插件退化 | 按键延迟 ≤50ms（P99），幽灵文字激活时同样满足 | 主线程不可被 AI 请求/渲染阻塞 |
| **NFR2** | AI 流式输出首 token 快速响应 | 首 token ≤1.5s（P95），含上下文组装 | 用户感知为"AI 在思考"而非"系统卡死" |
| **NFR3** | 上下文组装在可接受时间内完成 | ≤500ms（10 万字项目） | 多层记忆检索 + 压缩 + 组装总耗时 |
| **NFR4** | 大文档编辑器性能不退化 | 10 万字文档输入延迟 ≤50ms | TipTap 需分段加载或虚拟化策略 |
| **NFR5** | 自动保存不阻塞用户操作 | 保存操作 ≤2s（含 1s debounce），异步执行 | 用户无感知的后台保存 |
| **NFR6** | Dashboard 首屏快速加载 | LCP ≤2.5s | Server Components 优化 |
| **NFR7** | 首屏 JavaScript 体积可控 | ≤300KB gzipped | 编辑器和 AI 工具延迟加载 |

### Security

WriteTeam 处理两类敏感数据：用户手稿（知识产权）和 API Key（凭证信息）。

| NFR# | 需求 | 指标 | 说明 |
|------|------|------|------|
| **NFR8** | 用户数据完全隔离 | Row Level Security 覆盖所有表，零跨用户数据泄露 | Supabase RLS 策略：`user_id = auth.uid()` |
| **NFR9** | API Key 加密存储 | AES-256-GCM 加密，密钥不以明文出现在客户端或服务端日志 | `src/lib/ai/encryption.ts` |
| **NFR10** | AI 请求不经服务端存储用户手稿 | 服务端仅做 SSE 代理转发，不持久化用户发送给 LLM 的内容 | BYOK 模式下用户手稿直通 LLM Provider |
| **NFR11** | 认证会话安全刷新 | 每次请求刷新 session cookie，过期自动重定向登录 | `proxy.ts` + `updateSession()` |
| **NFR12** | 遵守中国《生成式AI管理办法》 | BYOK 模式将 AI 调用责任转移给用户；产品不提供内置模型 | 法律合规 |
| **NFR13** | 传输层加密 | 所有通信 HTTPS（TLS 1.2+） | Vercel + Supabase 默认启用 |

### Integration

BYOK 多模型集成是 WriteTeam 的核心架构特征。

| NFR# | 需求 | 指标 | 说明 |
|------|------|------|------|
| **NFR14** | 兼容所有 OpenAI-compatible API | 支持 ≥5 个 Provider（DeepSeek、OpenAI、Ollama、OpenRouter、硅基流动） | 通过统一 `/chat/completions` 端点 |
| **NFR15** | 流式协议稳定 | 流式响应中断后已接收内容保留率 100%，≤3s 内向用户展示错误状态并提供重试入口 | 网络不稳定场景下的韧性 |
| **NFR16** | 模型切换不影响功能完整性 | 故事意识引擎在不同模型上的功能完整性 ≥90% | 不同模型的上下文窗口差异需适配 |
| **NFR17** | AI 遥测记录完整 | 每次 AI 调用记录延迟、token 数、模型 ID 到 `ai_history` 表 | 用于性能监控和用量分析 |

### Reliability

用户手稿丢失是不可接受的灾难。

| NFR# | 需求 | 指标 | 说明 |
|------|------|------|------|
| **NFR18** | 文档内容不丢失 | 自动保存成功率 ≥99.9%，保存失败时本地缓存兜底 | 客户端 + 服务端双层保障 |
| **NFR19** | 误删数据可恢复 | Story Bible 条目删除后 7 天内可恢复（软删除） | 防误操作 |
| **NFR20** | 数据库备份 | Supabase 自动每日备份，可恢复到任意时间点 | Supabase Pro 特性 |
| **NFR21** | 离线/断网韧性 | 断网时编辑器内容不丢失（本地缓存），恢复后自动同步 | localStorage 或 IndexedDB 兜底 |

### Accessibility

| NFR# | 需求 | 指标 | 说明 |
|------|------|------|------|
| **NFR22** | 核心功能键盘可操作 | 所有主要功能可通过 Tab/Enter/Esc 完成 | WCAG 2.1 AA |
| **NFR23** | 颜色对比度达标 | 文本/背景对比度 ≥4.5:1（AA 标准） | Tailwind + shadcn/ui 默认满足 |
| **NFR24** | 支持文本缩放 | 200% 缩放不破坏布局 | WCAG 2.1 AA |

### Scalability

| NFR# | 需求 | 指标 | 说明 |
|------|------|------|------|
| **NFR25** | 单用户项目规模支撑 | 单项目支持 ≥100 万字（≥500 个文档） | 数据库查询和编辑器需优化 |
| **NFR26** | 并发用户基础支撑 | 初始支持 ≤1000 并发用户（Vercel + Supabase 默认能力） | 无需自建基础设施 |
