# WriteTeam 编辑器体验迭代设计（长篇小说作者）

日期：2026-03-03  
范围：一个月系统升级  
目标：在保持现有三栏架构、现有 AI 入口与 BYOK 隐私机制不变的前提下，提升日活写作时长。

## 1. 设计目标与约束

### 1.1 核心目标

- 降低写作过程中的打断频率，提升单次会话连续创作时间。
- 优先优化长篇小说作者在章节跳转、状态确认、连续写作中的体验。

### 1.2 明确约束

- 保持当前页面信息架构（现有三栏）不变。
- 保持现有 AI 工具入口（工具栏与选区菜单）不变。
- 保持 BYOK 与隐私机制不变，不新增密钥存储路径。

## 2. 方案选择结论

本次采用 `方案 A：写作流优先`，并吸收极少量高频 AI 快捷能力。

选择理由：

- 与“提升日活写作时长”目标直接对齐。
- 对既有结构侵入小，可在一个月内稳定落地。
- 在不增加认知负担的情况下提升流畅度与安全感。

## 3. 架构设计

### 3.1 总体策略

- 不重构主架构，保持 `EditorShell` 作为编辑器中枢。
- 通过“状态层 + 交互层增强”实现体验迭代。

### 3.2 新增能力边界

- 新增“写作会话状态层”（前端本地状态 + 轻量埋点）。
- 所有增强能力以可插拔子模块接入，不改变主数据链路。

## 4. 组件与交互设计

### 4.1 专注模式（Focus Mode）

- 一键进入专注写作，弱化两侧噪音信息。
- 保持快速召回能力（Esc/快捷键恢复），避免功能“消失感”。

### 4.2 章节导航增强

- 在左栏强化章节定位能力：最近编辑、字数进度、未完成标记。
- 减少长篇场景下的滚动查找成本。

### 4.3 写作安全感反馈

- 统一顶部轻提示：已保存 / 保存中 / 重试中 / 错误。
- 降低用户反复确认“是否丢稿”的心理负担。

### 4.4 高频 AI 快捷流

- 保留现有 AI 入口。
- 增加“最近一次 AI 操作一键复用”，减少重复配置与点击。

### 4.5 低打扰原则

- 新增反馈默认轻量可折叠。
- 避免弹窗式强打断。

## 5. 数据流与状态管理

### 5.1 主数据流保持不变

- 仍采用 TipTap -> autosave -> Supabase 的主链路。

### 5.2 会话态（本地）

- 管理项：专注状态、侧栏显示策略、当前章节锚点、最近 AI 操作。
- 使用 localStorage 持久化以支持刷新恢复。

### 5.3 保存状态聚合器

- 将分散保存信号聚合为统一 UI 状态：`saving` / `saved` / `retrying` / `error`。
- 所有保存反馈共享同一状态源，避免冲突提示。

### 5.4 AI 快捷复用的隐私边界

- 仅保存“操作类型 + 参数模板”。
- 不保存 API Key，不改变 BYOK 请求头传递路径。

### 5.5 度量埋点

- 记录交互与耗时事件（专注模式切换、章节切换耗时、AI 复用触发等）。
- 用于验证日活写作时长与中断率变化。

## 6. 错误处理与降级策略

- 保存错误统一在轻提示区呈现，并提供可执行动作（重试、恢复）。
- 新增模块均采用软失败：异常时回退至现有旧行为，不阻断写作。
- localStorage 读取失败时清空损坏状态并回退默认值。
- AI 复用失败时回退到现有工具栏手动流程。
- 采用按模块 feature flag 灰度与快速止损。

## 7. 测试与验收

### 7.1 测试范围

- 单测：会话状态管理、保存状态聚合器、AI 最近操作复用（含异常分支）。
- 组件测试：专注模式、章节导航、提示条状态切换与回退。
- 回归测试：三栏结构、AI 入口、BYOK 请求链路保持不变。

### 7.2 验收指标（灰度后 1-2 周）

- 人均单次写作会话时长提升。
- 写作过程中的无效切换/中断次数下降。
- 保存相关投诉率不上升。

### 7.3 发布策略

- 小流量灰度 -> 指标达标后全量。
- 若指标不达标，按模块关闭 feature flag 回滚。

## 8. 非目标（本轮不做）

- 不改造 AI 底层推理链路与模型接入方式。
- 不重构整体页面布局与导航体系。
- 不引入新的后端密钥管理机制。

## 9. 里程碑建议（一个月）

- 第 1 周：专注模式 + 保存状态聚合器。
- 第 2 周：章节导航增强 + 会话态持久化。
- 第 3 周：AI 最近操作复用 + 软失败与回退。
- 第 4 周：灰度发布、指标观察、调整与收敛。
