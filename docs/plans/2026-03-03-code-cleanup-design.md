# WriteTeam 保守代码清洁设计

**日期：** 2026-03-03  
**范围：** 全仓库（保守策略）

## 1. 清理边界与判定规则

### 1.1 目标
仅清理“可证明未使用”的生产代码与无用文件，降低维护负担，不改变业务行为。

### 1.2 允许操作
- 删除未被引用的模块文件、导出、常量、工具函数
- 删除可证明不可达分支
- 删除重复且未被使用的类型定义

### 1.3 禁止操作
- 不改动认证、权限与用户数据 ownership 约束
- 不下线任何 legacy 数据兼容路径
- 不删除旧测试文件
- 不进行结构性重构、目录迁移或功能改写

### 1.4 删除证据标准（必须同时满足）
- `rg` 全局检索无引用
- TypeScript/ESLint 无对该符号的使用需求
- 删除后通过 `npm run lint` 与 `npm run test`

## 2. 执行流程与变更批次策略

### 2.1 基线采集
- 记录 `git status --short`
- 执行 `npm run lint`
- 执行 `npm run test`

### 2.2 候选发现
- 使用 `rg --files src` 获取文件范围
- 对候选导出/文件执行全局检索，形成“候选项 + 证据”列表
- 不满足证据标准的项一律不纳入删除

### 2.3 小批次清理
- 每批处理 1-5 个候选项
- 每批仅做删除与最小必要引用修正
- 每批后运行受影响测试，异常时立即定位

### 2.4 批次验证
- 批次内：运行相关 Vitest 文件
- 批次结束：统一运行 `npm run lint` + `npm run test`

### 2.5 收尾核查
- 复查未触及 auth/ownership/legacy 禁区
- 产出清理清单（删除项、证据、验证结果）

## 3. 风险控制、回滚与验收定义

### 3.1 主要风险
- 动态引用误判（字符串路径、运行时注册）
- 测试覆盖盲区导致误删
- 与近期分支迭代产生语义冲突

### 3.2 控制措施
- 每个删除项都保留“引用证据 + 删除理由”
- 默认小批次推进，减少回归定位成本
- 对不确定项采用“保留并标注待确认”

### 3.3 回滚策略
- 任一批次失败，回退该批次改动，不扩散
- 保持改动粒度可单点撤销

### 3.4 验收标准
- 所有删除项均满足“可证明未使用”
- 未触碰 legacy 兼容逻辑、未删除旧测试
- `npm run lint` 与 `npm run test` 全量通过
- 交付可追溯清单（删除项 + 证据 + 验证结果）
