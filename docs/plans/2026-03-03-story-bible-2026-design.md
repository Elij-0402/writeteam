# WriteTeam 故事圣经 2026 全链路升级设计

日期：2026-03-03  
状态：已评审通过（待实施计划）  
目标：在保持现有三栏编辑器、BYOK 隐私边界与核心交互稳定的前提下，构建 AI Agent 时代的故事一致性最佳实践能力，并以“冲突率下降”为首要验收标准。

## 1. 设计目标与范围

### 1.1 核心目标

- 将“故事圣经”从静态资料面板升级为一致性中枢（Consistency Core）。
- 降低长篇写作中的人设漂移、世界规则冲突、时间线错位。
- 形成“生成前预检 -> 生成中约束 -> 生成后审校 -> 回填沉淀”的闭环。

### 1.2 本轮范围（全链路）

- Story Bible 结构升级（强兼容迁移）。
- 提示词上下文编排升级（任务感知与最小必要注入）。
- 连续性检查与修复工作流升级（可执行建议）。
- 指标与灰度机制升级（可观测、可回退）。

### 1.3 约束

- 保持现有编辑器主结构与核心入口不变。
- 不改变 BYOK 密钥处理路径，不新增服务端密钥持久化。
- 所有升级必须软失败可回退，不阻断写作。

## 2. 方案选择结论

采用：**双阶段“全链路一致性内核”方案**。

### 2.1 阶段划分

- 阶段 1（基础能力）：结构化知识层 + 提示词注入治理 + 基础冲突检测。
- 阶段 2（进阶能力）：主动预警、冲突评分、质量看板、策略优化闭环。

### 2.2 选择理由

- 与首要目标“冲突率下降”直接对齐。
- 满足“两阶段灰度”与“强兼容迁移”的实施要求。
- 相比一次性大重构，风险更低，价值释放更早。

## 3. 总体架构

在现有链路
`Story Bible -> buildStoryPromptContext -> AI Route -> continuity-check`
之上叠加 `Consistency Core`：

- 结构化知识层：可机器判定的事实与约束。
- 上下文编排层：按任务类型注入最小必要上下文。
- 校验反馈层：预检、审校、修复建议、知识回填。

该架构为增量式叠加，不替换现有主链路。

## 4. 组件与数据模型设计

### 4.1 新增核心对象

- `CanonFact`：不可违背事实（身份、能力边界、硬设定）。
- `TimelineEvent`：带时间锚点的关键事件（顺序、参与者、状态变化）。
- `CharacterArcState`：角色阶段状态（动机、关系、秘密进度）。
- `ConstraintRule`：写作约束（禁止项、必须项、风格红线）。

### 4.2 与现有字段关系

- 从 `story_bibles`（`synopsis/worldbuilding/notes/ai_rules/outline`）与 `characters` 提取初始结构化信息。
- 提取失败时保留原文并标记“待确认”，不影响旧链路使用。

### 4.3 前端能力分层

- 事实卡片层：结构化展示角色事实、世界规则、时间线。
- 冲突工作台层：展示冲突原因、证据来源、修复动作。
- 可见性策略层：在现有 visibility 基础上扩展到“按任务/按工具”粒度。

### 4.4 数据治理字段

每条结构化信息带：`source`（人工/AI推断）、`confidence`、`updated_at`，用于判定冲突优先级与可信度。

## 5. 数据流与 Agent 工作流

### 5.1 生成前（Preflight）

- 触发 AI 写作前执行轻量一致性预检。
- 命中高风险冲突时优先提示并给修复建议。

### 5.2 生成中（Context Assembly）

- 在 `buildStoryPromptContext` 中增加任务感知编排器。
- 仅注入当前任务/场景必需信息，降低噪声与冲突诱因。

### 5.3 生成后（Post-check）

- 增强 continuity-check 输出：结构化问题、证据链、`insert/replace` 修复动作。
- 支持“一键应用到正文”或“回填到故事圣经”。

### 5.4 多 Agent 协作职责

- Writer Agent：内容生成。
- Continuity Agent：规则核验。
- Librarian Agent：事实沉淀与圣经维护。

通过统一 `Consistency Core` 状态协作，避免各能力读取分散文本导致漂移。

## 6. 错误处理、降级与灰度

### 6.1 软失败与回退

- 结构化提取失败、预检超时、审校异常时自动回退旧逻辑。
- 不阻断编辑与生成主流程。

### 6.2 冲突分级

- `high`：硬规则冲突（默认阻断并要求确认）。
- `medium`：时间线/人设可疑（警告可继续）。
- `low`：风格偏移（仅提示）。

### 6.3 强兼容迁移

- 读路径：新结构优先，缺失回落旧字段。
- 写路径：阶段 1 双写（新结构 + 旧字段摘要），阶段 2 评估后收敛。
- 迁移异常：记录并重试，不影响用户当前编辑。

### 6.4 Feature Flag 策略

- `consistency_preflight`
- `structured_context`
- `post_check_enhanced`

三项可独立灰度与回滚，支持按用户/项目分批启用。

## 7. 测试与验收

### 7.1 测试分层

- 单测：提取器、冲突分级器、上下文编排器。
- 合同测试：审校输出 schema 与动作契约。
- 组件测试：冲突工作台、一键修复与回填。
- 回归测试：现有 Story Bible、AI 入口、BYOK 链路保持不变。

### 7.2 验收指标

- 主指标：连续性冲突率（每千字/每章）持续下降。
- 次指标：同类冲突复发率下降、修复建议采纳率上升。
- 守护指标：生成时延、误报率维持在可接受范围。

### 7.3 阶段门槛

- 阶段 1：冲突识别能力稳定可用，主指标出现下降趋势。
- 阶段 2：复发率与采纳率达到稳定改善。

## 8. 非目标

- 不重构编辑器三栏布局。
- 不改造 BYOK 密钥机制与服务端持久化策略。
- 不在本轮引入与一致性目标弱相关的重功能扩张。

## 9. 里程碑建议

- 第 1-2 周：结构化知识层、提取与回退机制。
- 第 3-4 周：上下文编排器、预检与增强审校输出。
- 第 5-6 周：冲突工作台、回填闭环、指标面板。
- 第 7-8 周：灰度发布、数据复盘、阈值调优。
