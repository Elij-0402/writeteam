# Dashboard 壳层 UX 对齐设计（降低迷失感）

**日期**: 2026-03-03  
**状态**: 已批准（首轮落地完成）  
**范围**: 全站壳层（`/dashboard`、`AppSidebar`、`SiteHeader`、空状态引导）

## 1. 目标与约束

### 目标
- 降低用户在壳层导航中的迷失感，让用户在进入应用后快速知道“下一步做什么”。
- 将 `/dashboard` 从静态欢迎页升级为“任务控制台”，优先支持“继续写作”。
- 在不改后端 schema 的前提下，统一壳层的信息层级和动作入口语义。

### 约束
- 允许调整信息架构（IA）与交互，不新增复杂后端能力。
- 保持现有项目/文档/编辑器核心能力不退化。
- 优先复用现有数据：`projects`、`documentsByProject`。

## 2. 方案选择

本轮评估了 3 条路线：
- A 任务驱动壳层（推荐并采用）
- B 引导驱动壳层（低风险，但结构问题缓解有限）
- C 极简模式壳层（认知负担低，但高频入口可能过深）

采用 A 的原因：它最直接服务“降低迷失感”目标，可在首屏明确主任务并缩短从进入到写作的路径。

## 3. 信息架构（IA）设计

### 3.1 Dashboard 定位
- `/dashboard` 改为任务控制台，而不是泛欢迎页。
- 中区固定三块：
  - `继续写作`（主按钮）
  - `最近文档`（3-5 条）
  - `下一步建议`（单条主建议）

### 3.2 左侧栏语义分层
- 全局区只保留全局动作（如“新建项目”）。
- 项目级动作集中在项目节点菜单（新建文档、编辑项目、打开画布、删除项目）。
- 文档级动作集中在文档节点菜单（重命名、上移/下移、删除）。

### 3.3 顶部状态强化
- `SiteHeader` 除面包屑外增加状态胶囊，解释当前上下文（如“未选择文档”“可继续”）。
- 避免“看得到界面但不知道下一步”的认知空窗。

### 3.4 条件化空状态
- 空状态按用户上下文分支展示，而非单模板：
  - 无项目
  - 有项目无文档
  - 有文档但未选中
  - 新用户首次进入

## 4. 组件与交互设计

### 4.1 Dashboard 组件
- `ContinueWritingCard`
  - 动作：跳转最近编辑文档
  - 文案：`继续写作`
- `RecentDocumentsList`
  - 展示：文档名、所属项目、更新时间
  - 点击：直接进入对应编辑页
- `NextActionCard`
  - 展示当前推荐动作
  - 动作一键执行（创建项目/创建首章/继续文档）

### 4.2 Sidebar 交互规则
- 创建或删除成功后，立即刷新壳层上下文并更新选中态。
- 错误统一反馈（toast + 局部提示），不做误导性乐观跳转。

### 4.3 Header 交互规则
- 状态胶囊与路由和选中文档同步。
- 非编辑态弱化 AI 入口优先级，避免主流程被侧向分流。

## 5. 数据流与状态决策

## 5.1 仅做壳层派生，不加后端复杂度
- 输入：`projects`、`documentsByProject`
- 输出（派生状态）：
  - `hasProjects`
  - `hasAnyDocuments`
  - `lastEditedDocument`
  - `recommendedNextAction`

### 5.2 推荐动作规则（YAGNI）
- 无项目 -> `create_project`
- 有项目无文档 -> `create_first_document`
- 有文档但当前未选中 -> `resume_last_document`
- 已在编辑文档 -> `continue_current_document`

### 5.3 导航与一致性
- “继续写作”始终指向 `lastEditedDocument`。
- 所有动作以成功回执为准再跳转，保证视图与数据一致。

## 6. 错误处理与反馈

- Action 返回 `{ error }` 时：
  - 保持当前上下文不跳转
  - 展示中文可行动错误信息
  - 局部组件显示失败状态（例如卡片内提示）
- 成功时：
  - 自动聚焦下一步目标（例如创建文档后自动选中并进入编辑）

## 7. 验收标准

### 可用性验收
- 新用户进入后 3 秒内可识别唯一主动作。
- 老用户进入 `/dashboard` 后 1 次点击可回到最近文档。
- 任意空状态都存在明确下一步动作，不出现“空白无出口”。

### 一致性验收
- 侧栏、头部状态、dashboard 卡片对“当前上下文”的表达一致。
- 创建/删除/重命名后的反馈、选中态、路由跳转保持一致。

## 8. 测试策略（最小集）

- Dashboard 状态分支渲染测试（4 类上下文）。
- Sidebar 动作可达性与反馈测试（全局/项目/文档三级）。
- Header 状态胶囊文案同步测试。
- 烟测：登录 -> dashboard -> 继续写作 -> editor。

## 9. 非目标

- 不引入复杂推荐算法。
- 不修改后端数据库 schema。
- 不重做 AI 面板能力，仅调整入口优先级与引导关系。
